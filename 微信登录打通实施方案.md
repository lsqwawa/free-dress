# 畅搭（FreeDress）App 与小程序登录体系打通实施方案

> 版本：v1.0  
> 日期：2026-05-29  
> 适用范围：`backend/`（NestJS + Prisma）、`FreeDressApp/`（React Native）、`freeDressWechat/`（微信小程序）

---

## 一、需求与目标

### 1.1 业务目标

1. **App 端**
   - 现有手机号 + 密码注册流程保留。
   - 注册/登录页新增"微信登录"入口，使用微信开放平台 SDK 拉起授权。
   - 用户首次微信登录后自动建号；后续可在「个人中心 → 账号绑定」中补充手机号、密码等信息。
2. **小程序端**
   - 现有微信一键登录保留（`wx.login` + 用户授权）。
   - 新增"手机号 + 密码"登录入口，复用 App 注册的账号。
   - 当用户用手机号登录后，若当前微信 OpenID 未与该账号绑定，且未冲突，**自动绑定**当前微信。
3. **跨端一致性**
   - 同一自然人无论从哪一端登录，最终对应同一 `User.id`。
   - 跨端识别依靠 **微信开放平台 UnionId**（要求 App 与小程序绑定到同一开放平台账号）。

### 1.2 非功能目标

- 平滑迁移：已存在的纯手机号用户不受影响。
- 安全：绑定 / 解绑、自动绑定均需防止账号被劫持。
- 可观测：所有微信登录、绑定、合并行为打点日志。

## 三、总体架构设计

### 3.1 跨端身份模型

```
                ┌────────── 微信开放平台账号（统一） ──────────┐
                │                                              │
   小程序 AppID                              移动应用 AppID
        │                                          │
   wx.login → code → jscode2session            微信 OpenSDK
   openid_mp + unionid                         openid_app + unionid
                │                                          │
                └─────────► UnionId（同一自然人） ◄────────┘
                                    │
                            后端 User 表统一识别
```

**核心原则**：以 `unionId` 为跨端用户主键；OpenID 用于补全 / 调起对应端能力。

### 3.2 用户身份组合

| 场景 | phone | password | wechatOpenIdApp | wechatOpenIdMp | unionId |
|------|-------|----------|------------------|-----------------|---------|
| 纯手机号注册（老用户） | ✅ | ✅ | — | — | — |
| App 微信首登 | — | — | ✅ | — | ✅ |
| 小程序微信首登 | — | — | — | ✅ | ✅ |
| 手机号 + 已绑微信 | ✅ | ✅ | 可选 | 可选 | ✅ |

## 四、数据库模型变更

### 4.1 `User` 模型扩展

```prisma
model User {
  id              String     @id @default(uuid())

  // 手机号体系（变为可选）
  phone           String?    @unique
  password        String?

  // 微信体系
  wechatUnionId   String?    @unique           // 跨端唯一标识
  wechatOpenIdApp String?    @unique           // App 端 OpenID
  wechatOpenIdMp  String?    @unique           // 小程序端 OpenID
  wechatNickname  String?
  wechatAvatarUrl String?

  // 基础资料
  nickname        String     @default("用户")
  avatarUrl       String?
  role            UserRole   @default(USER)

  // 注册来源（埋点 / 风控）
  registerSource  RegisterSource @default(PHONE)

  createdAt       DateTime   @default(now())
  updatedAt       DateTime   @updatedAt

  // 关联关系（保持不变）
  clothes         Cloth[]
  outfits         Outfit[]
  favorites       Favorite[]
  tryOnResults    TryOnResult[]

  @@map("users")
}

enum RegisterSource {
  PHONE         // 手机号注册
  WECHAT_APP    // App 微信登录
  WECHAT_MP     // 小程序微信登录
}
```

**关键约束**：
- `phone`、`wechatUnionId`、`wechatOpenIdApp`、`wechatOpenIdMp` 全部 `@unique`，但允许 `NULL`。Postgres 默认多个 NULL 不冲突，无需额外处理。
- 应用层校验：`phone + password` 必须成对出现（要么都有要么都没有）。

### 4.2 新增 `WechatBindLog`（可选 - 审计）

```prisma
model WechatBindLog {
  id        String   @id @default(uuid())
  userId    String
  action    String   // BIND_WECHAT | UNBIND_WECHAT | BIND_PHONE | MERGE
  platform  String   // APP | MP
  detail    String?  // JSON 字符串记录前后状态
  ip        String?
  createdAt DateTime @default(now())

  @@index([userId])
  @@map("wechat_bind_logs")
}
```

### 4.3 迁移策略

1. `prisma migrate dev --name add_wechat_login` 生成迁移：
   - `phone` / `password` 变为 nullable（Postgres `ALTER COLUMN ... DROP NOT NULL`）。
   - 新增微信字段、`registerSource` 枚举。
2. **数据回填**：执行 `UPDATE users SET register_source = 'PHONE' WHERE register_source IS NULL;`（迁移自动处理 default）。
3. 上线灰度：保持现有手机号注册逻辑兼容，仅新增微信能力。

## 五、后端 API 设计

### 5.1 新增微信登录接口

#### 5.1.1 `POST /auth/wechat/mp-login` —— 小程序登录

请求：
```json
{ "code": "wx.login() 返回的 code", "userInfo": { "nickName": "...", "avatarUrl": "..." } }
```

流程：
1. 调用微信开放接口 `https://api.weixin.qq.com/sns/jscode2session`，取得 `openid`、`unionid`、`session_key`。
2. 查用户：
   - 优先按 `wechatUnionId` 匹配。
   - 次按 `wechatOpenIdMp` 匹配。
3. 若命中 → 更新 `wechatOpenIdMp / unionId`（兜底），返回 token。
4. 未命中 → 创建新 `User`：`registerSource=WECHAT_MP`，写入 OpenID / UnionId / 微信昵称 / 头像。
5. 返回 `{ accessToken, refreshToken, user, needBindPhone: !user.phone }`。

#### 5.1.2 `POST /auth/wechat/app-login` —— App 登录

请求：
```json
{ "code": "微信 OpenSDK 授权返回的 code" }
```

流程：
1. 调用 `https://api.weixin.qq.com/sns/oauth2/access_token?appid=&secret=&code=&grant_type=authorization_code` 拿 `access_token` + `openid` + `unionid`。
2. 调用 `https://api.weixin.qq.com/sns/userinfo` 拿头像 / 昵称（可选）。
3. 查询 / 创建用户逻辑同 MP（区别：`wechatOpenIdApp`、`registerSource=WECHAT_APP`）。
4. 返回 `{ accessToken, refreshToken, user, needBindPhone }`。

### 5.2 账号绑定接口（需登录态）

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/auth/bind/phone` | 已登录账号绑定手机号（验证短信/图形验证码 + 设置密码） |
| `POST` | `/auth/bind/wechat-mp` | 已登录账号绑定小程序微信（传 `code`） |
| `POST` | `/auth/bind/wechat-app` | 已登录账号绑定 App 微信（传 `code`） |
| `POST` | `/auth/unbind/wechat` | 解绑微信（要求账号已绑定手机号，避免无凭证锁死） |

#### 5.2.1 `POST /auth/bind/phone` 关键校验

- 当前 token 解析出的 `userId`。
- `phone` 必须未被其他用户占用，否则返回 `409 PHONE_ALREADY_USED`。
- 校验图形/短信验证码。
- 写入 `phone + password`，更新 `updatedAt`。

#### 5.2.2 微信绑定关键校验

- 取出 `openid + unionid`。
- 若 `unionid` 已被其他账号占用：
  - 若当前账号未绑过手机号且数据为空 → 走 **账号合并**（5.3 节）。
  - 否则返回 `409 WECHAT_ALREADY_BOUND`，提示用户先解绑。

### 5.3 自动绑定与账号合并

#### 5.3.1 小程序手机号登录 + 自动绑微信

`POST /auth/login`（手机号）调用流程在 **小程序场景** 增加一段：

```
请求体：{ phone, password, wechatCode?: string }   ← 小程序登录页带上 wx.login 的 code
```

后端逻辑（在原有 login 成功后）：
```
if (req.wechatCode) {
  const { openid, unionid } = await jscode2session(req.wechatCode);

  const existing = await db.user.findFirst({
    where: { OR: [{ wechatUnionId: unionid }, { wechatOpenIdMp: openid }] }
  });

  if (!existing) {
    // 当前微信未被绑定 → 自动绑到该手机号账号
    await db.user.update({
      where: { id: user.id },
      data: { wechatOpenIdMp: openid, wechatUnionId: unionid }
    });
    log('AUTO_BIND_WECHAT_MP');
  } else if (existing.id !== user.id) {
    // 微信已绑别的账号 → 不强行抢绑，返回提示
    response.autoBindResult = 'CONFLICT';
  }
}
```

#### 5.3.2 账号合并（高级，二期可选）

若用户在 App 端有「微信号」账号、又在小程序端用手机号登录并触发了"微信已被另一账号占用"，提供一次性的「账号合并」流程：
- 后端：`POST /auth/merge`（需校验两端身份）
- 数据迁移：将 `Cloth / Outfit / Favorite / TryOnResult / AiQuota` 从被合并账号迁移到目标账号，删除被合并账号或标记 `mergedInto`。

> **一期不实现**，仅返回冲突提示，引导用户自行解绑后重新绑定。

### 5.4 JWT Payload 调整

- 旧：`{ sub, phone }`
- 新：`{ sub, phone?: string, openIdMp?: string, openIdApp?: string }`
- 业务依赖 `phone` 的地方改为依赖 `sub`（`userId`）。

### 5.5 错误码

| Code | HTTP | 含义 |
|------|------|------|
| `PHONE_ALREADY_USED` | 409 | 绑定的手机号已被他人占用 |
| `WECHAT_ALREADY_BOUND` | 409 | 微信已被他人绑定 |
| `WECHAT_CODE_INVALID` | 400 | 微信 code 无效或过期 |
| `BIND_LAST_AUTH_FORBIDDEN` | 400 | 解绑后账号将无任何登录方式 |

## 七、安全与边界情况

| 风险 | 防御策略 |
|------|----------|
| 微信 code 重放 | code 仅一次有效，单次调用 `jscode2session` 即失效 |
| 微信账号被冒用绑定 | 绑定前必须校验当前登录态（JWT）；UnionId 冲突拒绝 |
| 解绑后无法登录 | 解绑微信前校验账号必须存在手机号 + 密码；同理解绑手机号需有微信 |
| 自动绑定误绑 | 仅在「当前微信未被任何账号绑定」时自动绑；冲突仅提示，不改数据 |
| 手机号被抢绑 | 绑定手机号需短信码（一期可降级为图形验证码 + 后台限频） |
| Token 旧版兼容 | 旧 JWT 仅含 `{ sub, phone }`，逻辑改为以 `sub` 为主；下发新 token 即可 |
| 多设备并发 | RefreshToken 单设备一份，绑定操作走幂等校验 |

## 十、验收标准

1. **App 端**
   - 新用户通过微信一键登录可直接进入 App，资料区显示「未绑定手机号」。
   - 在「账号与安全」中绑定手机号 + 密码成功，下次卸载重装后用手机号也可登录到同一账号。
   - 解绑微信前若未绑手机号，按钮置灰或弹提示。
2. **小程序端**
   - 新用户通过微信登录直接进入小程序。
   - 已在 App 端注册手机号的用户，在小程序选择「手机号登录」并输入正确凭证后：
     - 当前微信未绑定 → 自动绑定，下次微信打开小程序无需再次输入密码。
     - 当前微信已绑定到其他账号 → 提示用户冲突，不破坏数据。
3. **跨端**
   - App 微信登录 + 小程序微信登录 → 同一 `User.id`（UnionId 命中）。
   - 资料修改在两端实时同步（依赖现有 `GET /users/profile`）。
4. **回归**
   - 旧手机号 + 密码用户登录 / 注册 / 改密 / 找回密码 / 试穿 / 推荐功能全部正常。

## 十一、风险与回滚

- **风险**：微信开放平台审核未通过，导致 UnionId 不下发 → 退化为「OpenID 分端管理」，跨端识别失败。
  - 缓解：上线前用沙箱账号端到端验证 UnionId 字段。
- **风险**：迁移把 `phone NOT NULL` 改为 nullable 后，旧业务代码空指针。
  - 缓解：搜索后端所有 `user.phone` 使用点，做 `?.` 兜底；测试覆盖。
- **回滚**：保留迁移回滚脚本；前端隐藏微信入口即可降级。

### 小程序手机号登录页提交逻辑

```js
// pages/phoneLogin/phoneLogin.js
async handleLogin() {
  const { code } = await wx.login();
  const res = await authApi.phoneLogin({
    phone: this.data.phone,
    password: this.data.password,
    wechatCode: code,
  });
  app.setAuth(res.data);
  if (res.data.autoBindResult === 'CONFLICT') {
    wx.showToast({ title: '当前微信已绑定其他账号', icon: 'none' });
  }
  wx.switchTab({ url: '/pages/home/home' });
}
```

### 后端自动绑定核心逻辑（伪代码）

```ts
async login(dto: LoginDto, ctx: { ip: string }) {
  const user = await this.validateCredentials(dto.phone, dto.password);
  let autoBindResult: 'OK' | 'CONFLICT' | undefined;

  if (dto.wechatCode) {
    const { openid, unionid } = await this.wechat.jscode2session(dto.wechatCode);
    const conflict = await this.prisma.user.findFirst({
      where: {
        OR: [{ wechatUnionId: unionid }, { wechatOpenIdMp: openid }],
        NOT: { id: user.id },
      },
    });

    if (conflict) {
      autoBindResult = 'CONFLICT';
    } else if (!user.wechatUnionId) {
      await this.prisma.user.update({
        where: { id: user.id },
        data: { wechatUnionId: unionid, wechatOpenIdMp: openid },
      });
      autoBindResult = 'OK';
    }
  }

  return { ...await this.generateTokens(user.id), user, autoBindResult };
}
```
